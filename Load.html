2. Analiza e Rrjedhës së Paketave sipas Modeve të Rrjetit

Më poshtë paraqiten të gjitha diagramet e topologjive dhe konfigurimi i saktë i paketa-header-ave për secilën metodë gjatë të gjitha fazave të rrugëtimit (Klient -> Load Balancer -> Real Server -> Klient).

2.1 Layer 4 DR Mode (Direct Routing) - One Arm

Në modin Layer 4 DR, Load Balancer ndryshon vetëm adresën MAC të destinacionit për të përputhur me atë të Real Server (L2), pa e prekur adresën IP (L3). Serveri Real duhet të ketë VIP në loopback dhe i përgjigjet klientit direkt duke anashkaluar LB.

Struktura e Paketave (IP/MAC Header) për DR Mode:

Hapi dhe Rruga Burimi IP (Src IP) Dest IP (Dst IP) Burimi MAC (Src MAC) Dest MAC (Dst MAC)
1. Klienti -> Ruter 198.51.100.50 192.168.1.100 (VIP) 00:aa:bb:cc:dd:01 (Client) 00:aa:bb:cc:dd:02 (Router)
2. Ruter -> Load Balancer 198.51.100.50 192.168.1.100 (VIP) 00:aa:bb:cc:dd:02 (Router) 00:aa:bb:cc:dd:99 (LB_MAC)
3. LB -> Real Server 1 (L2 Ndryshim) 198.51.100.50 192.168.1.100 (VIP) 00:aa:bb:cc:dd:99 (LB_MAC) 00:aa:ee:ee:ee:10 (RS1_MAC)
4. RS1 -> Ruter (Kthim i drejtpërdrejtë) 192.168.1.100 (VIP) 198.51.100.50 00:aa:ee:ee:ee:10 (RS1_MAC) 00:aa:bb:cc:dd:02 (Router)

2.2 Layer 4 NAT Mode (Network Address Translation)

Në modin Layer 4 NAT, Load Balancer vepron si Gateway. Kur paketa vjen te LB, destinacioni IP ndryshohet nga VIP në RIP e serverit real (DNAT), ndërsa adresa e burimit mbetet ajo e Klientit. Serverat realë duhet të kenë si Default Gateway vetë Load Balancer-in në mënyrë që trafiku i kthimit të kalojë sërish përmes LB për të zhbërë NAT-in.

Struktura e Paketave (IP/MAC Header) për NAT Two-Arm Mode:

Fazë dhe Rrugëtimi Burimi IP (Src IP) Dest IP (Dst IP) Burimi MAC (Src MAC) Dest MAC (Dst MAC)
1. Klienti -> LB (VIP) 198.51.100.50 198.51.100.1 (VIP) 00:aa:bb:cc:dd:01 (Client) 00:aa:bb:cc:dd:99 (LB_Ext)
2. LB -> RS1 (DNAT i kryer) 198.51.100.50 192.168.1.10 (RIP1) 00:aa:bb:cc:dd:98 (LB_Int) 00:aa:ee:ee:ee:10 (RS1_MAC)
3. RS1 -> LB (Përmes GW) 192.168.1.10 (RIP1) 198.51.100.50 00:aa:ee:ee:ee:10 (RS1_MAC) 00:aa:bb:cc:dd:98 (LB_Int)
4. LB -> Klient (Zhbërje NAT) 198.51.100.1 (VIP) 198.51.100.50 00:aa:bb:cc:dd:99 (LB_Ext) 00:aa:bb:cc:dd:01 (Client)

2.3 Layer 4 SNAT Mode (Source Network Address Translation)

Në modin Layer 4 SNAT, Load Balancer ndryshon njëkohësisht edhe adresën IP të burimit (nga Client_IP në IP-në e brendshme të LB - LIP) dhe destinacionin IP (nga VIP në RIP). Kjo eliminon nevojën për ndryshimin e Default Gateway në Real Server, por e bën serverin të mos e shohë dot IP-në reale të klientit në shtresën e rrjetit.

Struktura e Paketave (IP/MAC Header) për SNAT One-Arm:

Fazë dhe Rrugëtimi Burimi IP (Src IP) Dest IP (Dst IP) Burimi MAC (Src MAC) Dest MAC (Dst MAC)
1. Klienti -> LB (VIP) 198.51.100.50 192.168.1.100 (VIP) 00:aa:bb:cc:dd:01 (Client) 00:aa:bb:cc:dd:99 (LB_MAC)
2. LB -> RS1 (SNAT & DNAT) 192.168.1.99 (LIP) 192.168.1.10 (RIP1) 00:aa:bb:cc:dd:99 (LB_MAC) 00:aa:ee:ee:ee:10 (RS1_MAC)
3. RS1 -> LB (Kthim i thjeshtë LAN) 192.168.1.10 (RIP1) 192.168.1.99 (LIP) 00:aa:ee:ee:ee:10 (RS1_MAC) 00:aa:bb:cc:dd:99 (LB_MAC)
4. LB -> Klient 192.168.1.100 (VIP) 198.51.100.50 00:aa:bb:cc:dd:99 (LB_MAC) 00:aa:bb:cc:dd:01 (Client)

2.4 Layer 7 SNAT Mode (Application Proxy)

Në Layer 7 SNAT, Load Balancer ndërpret plotësisht lidhjen TCP të klientit (Full Proxy), kryen analizimin e shtresës së aplikacionit (cookies, SSL terminim, rishkrim URL-sh), dhe më pas hap një lidhje krejtësisht të re TCP me Real Server. Ndarja fizike e sesioneve rrit sigurinë dhe fleksibilitetin, por kërkon më shumë CPU dhe memorie nga LB.

Rrjedha e Sesioneve TCP dhe Headers (L7):

Sesioni TCP Dërguesi ➔ Marrësi Burimi IP (Src IP) Dest IP (Dst IP) Shtresa 7 (HTTP Header)
Lidhja TCP 1 (Klient <-> LB) Klient ➔ LB (Kërkesa) 198.51.100.50 192.168.1.100 (VIP) GET /index.html HTTP/1.1
Lidhja TCP 1 (Klient <-> LB) LB ➔ Klient (Përgjigje) 192.168.1.100 (VIP) 198.51.100.50 HTTP/1.1 200 OK
Lidhja TCP 2 (LB <-> Server Real) LB ➔ RS1 (Kërkesë e re) 192.168.1.99 (LIP) 192.168.1.10 (RIP1) GET /index.html
X-Forwarded-For: 198.51.100.50
Lidhja TCP 2 (LB <-> Server Real) RS1 ➔ LB (Përgjigje) 192.168.1.10 (RIP1) 192.168.1.99 (LIP) HTTP/1.1 200 OK

Layer 4 DR Mode (Direct Routing)

Ultra-Fast

Në modin Layer 4 DR, Load Balancer (LB) vepron plotësisht në shtresën e dytë (Layer 2 - Data Link). LB pranon kërkesat e drejtuara nga VIP dhe pa ndryshuar IP-në e burimit ose destinacionit, modifikon adresën MAC të destinacionit duke vendosur atë të një prej Real Servers.

  • Përgjigjja anashkalon plotësisht LB-në duke u dërguar direkt te Klienti.
  • Ofron performancën më të lartë të mundshme në rrjet.
  • Kërkon që Real Serverët të kenë të konfiguruar VIP-in në Loopback.
  • Sfidë kritike: Zgjidhja e problemit ARP për të shmangur konfliktet IP.

Layer 4 NAT Mode

Transparent

Në modin Layer 4 NAT, LB kryen Destination NAT (DNAT). Ndryshon adresën e destinacionit nga VIP në RIP (Real IP) të serverit. Burimi (Client IP) mbetet i pandryshuar. Real Serveri duhet patjetër të ketë Default Gateway adresën IP të brendshme të LB.

  • Serveri Real sheh IP-në e vërtetë të klientit pa asnjë header shtesë.
  • Mbron Real Serverët në një subnet privat të izoluar.
  • Trafiku i kthimit duhet të kalojë detyrimisht përmes LB.
  • Kërkon ndryshim të tabelës së rrugëtimit (routing/GW) në server.

Layer 4 SNAT Mode

No Server Changes

Në modin Layer 4 SNAT, LB kryen njëkohësisht Source NAT dhe Destination NAT. Ndryshon destinacionin nga VIP në RIP dhe burimin nga Client IP në IP-në e brendshme të vetë LB (LIP).

  • Nuk kërkohet asnjë ndryshim rrugëzimi apo konfigurimi në Real Server.
  • Integrohet lehtësisht në rrjete ekzistuese (One-Arm).
  • Jo-transparent: Serverët nuk e shohin IP-në e klientit (shehin vetëm LB).
  • Përpunim shtesë në LB për dy modifikime IP për çdo paketë.

Layer 7 SNAT Mode

Application Proxy

Load Balancer funksionon si Full Reverse Proxy (p.sh. HAProxy/Nginx). Ndërpret lidhjen TCP me klientin, analizon kërkesën HTTP/HTTPS dhe hap një lidhje të re TCP të dytë me Real Server.

  • Mbështet Sticky Sessions, SSL Termination dhe URL Rewrite.
  • Siguri e shtuar duke ndarë tërësisht sesionet e jashtme me ato të brendshme.
  • Konsumon më shumë CPU/RAM; ka performancën më të ulët nga të katër modet.
  • Kërkon proxy protokol ose X-Forwarded-For për transparencë IP.

Sfidë Kritike: "Problemi ARP" (The ARP Problem) në DR Mode

Meqenëse në Direct Routing (DR), të gjithë Real Serverët kanë të njëjtën adresë VIP të konfiguruar në loopback (në mënyrë që të pranojnë paketat e dërguara nga LB), nëse ata i përgjigjen kërkesave ARP të ruterave ose pajisjeve të rrjetit për atë VIP, do të shkaktohej konflikt i tmerrshëm IP adresash. Rrjeti do të bëhej i paqëndrueshëm pasi kërkesat ARP do të shkonin në mënyrë të rastësishme te serverët në vend që të shkonin vetëm te Load Balancer.

ZGJIDHJA: Duhet të konfigurohen parametrat e kernelit Linux në Real Serverët (në /etc/sysctl.conf) që të mos i përgjigjen ARP kërkesave për VIP-in në loopback dhe të mos e njoftojnë atë në rrjet:

3. Konfigurimet e Serverit & Shembuj Kodi

Konfigurimi aktiv: /etc/sysctl.conf (Linux) 
# Konfigurimi për zgjidhjen e problemit ARP në Real Servers (L4 DR Mode)
# Shto këto rreshta në fund të skedarit /etc/sysctl.conf

net.ipv4.conf.all.arp_ignore = 1
net.ipv4.conf.all.arp_announce = 2
net.ipv4.conf.lo.arp_ignore = 1
net.ipv4.conf.lo.arp_announce = 2

# Pasi t'i ruani, aplikoni ndryshimet me komandën:
# sysctl -p

Ky konfigurim ndalon kernelin Linux që t'u përgjigjet kërkesave ARP ose të njoftojë adresën VIP të konfiguruar në loopback, duke lejuar vetëm Load Balancer-in t'u përgjigjet kërkesave ARP të rrjetit për këtë IP.
